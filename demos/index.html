Informática en la nube:

Significa que los usuarios pueden acceder a software y aplicaciones desde cualquier punto.

Modelo de responsabilidad compartida:

La seguridad de la nube es responsabilidad del CSP, y la seguridad en la nube es responsabilidad del cliente.

Modelos de nube, incluidos públicos, privados e híbridos:

Los modelos de nube son las diferentes formas en que se implementan y gestionan los servicios de computación en la nube.

Casos de uso adecuados para cada modelo de nube:

Nube Pública Nube Privada Nube Híbrida
Casos de uso Desarrollo y prueba de aplicaciones. Sitios web y aplicaciones de bajo costo. Datos y aplicaciones sensibles que requieren control total. Cargas de trabajo críticas para el negocio. Cargas de trabajo que requieren flexibilidad y escalabilidad. Respaldo y recuperación de desastres.

[AZ-900] Descripción de la arquitectura y los servicios de Azure.

Ian Castello. Enero 16, 2024.

1. Descripción de los principales componentes arquitectónicos de Azure

1.1 Describir las regiones de Azure, los pares de regiones y las regiones soberanas

Azure tiene regiones globales para usuarios comunes, pares de regiones para redundancia, regiones soberanas para entidades gubernamentales y regiones de nube nacional para cumplir con regulaciones locales. Esto asegura disponibilidad, recuperación ante desastres y cumplimiento normativo.

1.2 Describir las zonas de disponibilidad

Las Zonas de Disponibilidad en Azure son ubicaciones separadas para mejorar la disponibilidad y prevenir fallos. Permiten distribuir cargas de trabajo y cuentan con centros de datos independientes. Desplegar recursos en diferentes zonas crea una arquitectura resistente a fallos.

1.3 Describir los centros de datos de Azure

Los centros de datos de Azure son instalaciones globales eficientes y seguras que alojan la infraestructura de la nube. Ofrecen redundancia y conectividad rápida, asegurando servicios confiables y cercanía a los usuarios

1.4 Descripción de los recursos y grupos de recursos de Azure

En Azure, los recursos son elementos individuales (como máquinas virtuales), y los grupos de recursos son contenedores que los organizan para facilitar la administración.

1.5 Describir las suscripciones

En Azure, una suscripción es un plan de pago que organiza y permite el acceso a servicios en la nube.

1.6 Describir los grupos de administración

Los Grupos de Administración en Azure son contenedores que organizan suscripciones para una gestión eficiente.

1.7 Describir la jerarquía de grupos de recursos, suscripciones y grupos de administración

En Azure, los recursos están en Grupos de Recursos, estos están en Suscripciones, y varias Suscripciones se pueden agrupar en Grupos de Administración para una gestión eficiente.

2. Descripción de los servicios de proceso y redes de Azure

2.1 Comparación de los tipos de proceso
  • Contenedores para portabilidad
  • Máquinas virtuales para control total
  • Funciones para ejecutar código sin preocuparse por la infraestructura

La elección depende de la carga de trabajo y las necesidades de la aplicación.

2.2 Descripción de las opciones de máquina virtual
  • Azure Virtual Machines: Máquinas virtuales personalizables para ejecutar aplicaciones y sistemas operativos.
  • Azure Virtual Machine Scale Sets: Conjunto de máquinas virtuales idénticas que escala automáticamente para satisfacer la demanda
  • Conjuntos de Disponibilidad: Agrupación de VMs para garantizar alta disponibilidad y resistencia ante fallos.
  • Azure Virtual Desktop: Ofrece máquinas virtuales en la nube para acceder a escritorios y aplicaciones desde cualquier lugar.
2.3 Descripción de los recursos necesarios para las máquinas virtuales

Recursos clave para máquinas virtuales en Azure: CPU, RAM, almacenamiento, red, sistema operativo, IP y reglas de seguridad. Ajustar según necesidades específicas de la carga de trabajo.

2.4 Descripción de las opciones de hospedaje de aplicaciones
  • Aplicaciones Web: Fácil y escalable.
  • Contenedores: Uso de entornos contenerizados con servicios como AKS
  • Máquinas Virtuales: Flexibilidad total para personalizar entornos.
2.5 Descripción de las redes virtuales
  • Azure Virtual Networks:Conecta recursos de Azure de forma segura y organiza la red en segmentos aislados.
  • Subredes Virtuales de Azure: Divisiones lógicas dentro de una red virtual para organizar recursos.
  • Emparejamiento: Conecta redes virtuales para facilitar la comunicación entre recursos.
  • Azure DNS:Resuelve nombres de dominio a direcciones IP para aplicaciones en Azure.
  • VPN Gateway: Establece conexiones seguras entre la red virtual de Azure y la infraestructura local.
  • ExpressRoute: Proporciona conexiones dedicadas y de alto rendimiento entre Azure y la infraestructura local.
2.6 Definir puntos de conexión públicos y privados

Los puntos de conexión públicos son para acceso desde Internet con direcciones IP públicas, mientras que los privados son solo para redes internas con direcciones IP privadas. Se configuran según seguridad y necesidades de acceso.

3. Descripción de los servicios de almacenamiento de Azure

3.1 Comparación de los servicios de almacenamiento de Azure
Servicio Descripción Características Clave
Azure Blob Storage. Almacenamiento de objetos para datos no estructurados Alta disponibilidad, escalabilidad, almacenamiento en frío
Azure Table Storage Base de datos NoSQL para almacenamiento clave-valor Escalabilidad horizontal, acceso rápido mediante claves
Azure File Storage Almacenamiento en la nube accesible como recurso compartido de archivos Compartición de datos entre instancias de máquinas virtuales
Azure Queue Storage Servicio de cola simple y duradero para comunicación asíncrona Mensajería asíncrona, desacoplamiento, escalabilida
3.2 Descripción de los niveles de almacenamiento

  1. Caliente:Para datos frecuentemente accedidos, con mayor costo y baja latencia.
  2. Frío:Para datos con acceso menos frecuente, más económico que el almacenamiento en caliente.
  3. Archivado:El nivel más barato, para datos raramente accedidos, con mayor latencia.
3.3 Descripción de las opciones de redundancia

  • LRS:Redundancia local en un centro de datos.
  • ZRS:Redundancia en múltiples zonas para mayor disponibilidad.
  • GRS:Redundancia geográfica con copia asíncrona en otra región
  • RA-GRS:Similar a GRS, pero con acceso de lectura a la región secundaria.
  • LRS Premium:Redundancia local para discos Premium con alto rendimiento.
3.4 Descripción de las opciones de la cuenta de almacenamiento y los tipos de almacenamiento

Al crear una cuenta de almacenamiento en Azure, decides cómo replicar datos, permitir acceso público, habilitar CDN, configurar seguridad, retención de logs, controlar la transferencia de datos, generar firmas de acceso, establecer alertas y versiones de blobs.

3.5 Identificación de opciones para mover archivos, incluidos AzCopy, Explorador de Azure Storage y Azure File Sync
  1. AzCopy:Herramienta de línea de comandos para transferencias rápidas y masivas.
  2. Explorador de Azure Storage: Interfaz gráfica fácil de usar para transferencias manuales.
  3. Azure File Sync:Sincronización de archivos entre servidores locales y Azure.
  4. Azure Data Factory: Orquestación de flujos de trabajo para transferencia y manipulación de datos.
  5. Azure Storage Explorer:Aplicación de escritorio intuitiva para gestionar y transferir datos.
  6. Azure PowerShell y Azure CLI:Interfaces de línea de comandos para automatización y scripts.
3.6 Descripción de las opciones de migración, incluidas Azure Migrate y Azure Data Box
  • Azure Migrate:Evalúa, migra y optimiza activos locales a Azure.
  • Azure Site Recovery:Replica y recupera máquinas virtuales y servidores a Azure.
  • Azure Database Migration Service:Facilita la migración de bases de datos locales a servicios en la nube.
  • Azure Data Box:Facilita la migración de bases de datos locales a servicios en la nube.
  • Azure Data Box EdgeSimilar a Data Box, con capacidades de procesamiento en el borde.
  • Rehost (Levantar y Migrar):Mueve aplicaciones existentes a Azure sin modificar.
  • Refactor (Reescribir):Modifica aplicaciones para aprovechar servicios nativos en la nube.
  • Re-architect (Re-Arquitectar)Rediseña completamente aplicaciones para aprovechar al máximo la nube.

RBAC en Azure permite gestionar el acceso asignando roles a usuarios. Con roles predefinidos y asignación dinámica, se integra con Azure AD. Ofrece seguridad mejorada, facilita el cumplimiento normativo y simplifica la administración de accesos en Azure.

Configuración del acceso al almacenamiento

Configuración de redes virtuales y firewalls de Azure Storage

La configuración de redes virtuales y firewalls de Azure Storage es una parte importante de la seguridad de los datos en la nube. Azure Storage proporciona un modelo de seguridad por capas que le permite controlar el nivel de acceso a las cuentas de almacenamiento que exigen sus aplicaciones y entornos empresariales, en función del tipo y el subconjunto de redes o recursos que usa.

Para configurar el acceso de red a Azure Storage, puede limitar el acceso a su cuenta de almacenamiento a las solicitudes procedentes de direcciones IP especificadas, intervalos IP, subredes en una red virtual de Azure o instancias de recursos de algunos servicios de Azure. Las cuentas de almacenamiento tienen un punto de conexión público accesible a través de Internet. También puede crear puntos de conexión privados para la cuenta de almacenamiento. La creación de puntos de conexión privados asigna una dirección IP privada desde la red virtual a la cuenta de almacenamiento. Esto ayuda a proteger el tráfico entre la red virtual y la cuenta de almacenamiento a través de un vínculo privado.

El firewall de Azure Storage proporciona control de acceso para el punto de conexión público de la cuenta de almacenamiento. También puede usar el firewall para bloquear todo el acceso a través del punto de conexión público al usar puntos de conexión privados. La activación de las reglas de firewall para la cuenta de almacenamiento bloquea las solicitudes entrantes para los datos de forma predeterminada, a menos que las solicitudes procedan de un servicio que funcione en una instancia de red virtual de Azure o desde direcciones IP públicas permitidas.

Los tokens de SAS de delegación de usuarios se protegen con credenciales de Microsoft Entra. Un token de SAS proporciona acceso delegado y seguro a los recursos de la cuenta de almacenamiento de Azure.

En general, los tokens de SAS funcionan así:

  • La aplicación envía el token de SAS a Azure Storage como parte de una solicitud de API REST.
  • Si el servicio de almacenamiento confirma que la firma SAS es válida, la solicitud se autoriza.
  • Si el token de SAS no se considera válido, la solicitud se rechaza con el código de error 403 (prohibido).

En cuanto a la configuración de la cuenta de almacenamiento, puedes establecer la configuración de redundancia, el nivel de rendimiento y otras opciones.

Configuración de la redundancia de Azure Storage

La redundancia en Azure Storage se refiere a la capacidad de almacenar múltiples copias de los datos para protegerlos de eventos planificados y no planificados, como fallas de hardware transitorias, cortes de energía o de red y desastres naturales. La redundancia asegura que tu cuenta de almacenamiento cumpla con sus objetivos de disponibilidad y durabilidad incluso en caso de fallas.

En cuanto a la configuración de la redundancia de Azure Storage, hay varias opciones disponibles. En la región primaria, los datos se replican tres veces de forma síncrona en una sola ubicación física. Esta opción se llama Locally Redundant Storage (LRS) y es la opción de replicación más económica, pero no se recomienda para aplicaciones que requieren alta disponibilidad o durabilidad.

Otra opción es Zone-redundant storage (ZRS), que replica los datos de forma síncrona en tres zonas de disponibilidad de Azure en la región primaria. Microsoft recomienda usar ZRS en la región primaria para aplicaciones que requieren alta disponibilidad y también replicar los datos en una región secundaria.

Además, Azure Storage ofrece la opción de Geo-redundant storage (GRS), que replica los datos de forma síncrona tres veces en la región primaria y de forma asíncrona en una región secundaria geográficamente distante. Si la región primaria falla, los datos se pueden recuperar de la región secundaria.

Por último, Azure Storage también ofrece la opción de Read-access geo-redundant storage (RA-GRS), que replica los datos de forma síncrona en la región primaria y de forma asíncrona en una región secundaria geográficamente distante. Además, permite el acceso de lectura a los datos replicados en la región secundaria si la región primaria no está disponible por cualquier motivo.

Configuración de la replicación de objetos para blobs en bloques

La replicación de objetos en Azure Storage se refiere a la capacidad de almacenar múltiples copias de los datos para protegerlos de eventos planificados y no planificados, como fallas de hardware transitorias, cortes de energía o de red y desastres naturales. La replicación asegura que tu cuenta de almacenamiento cumpla con sus objetivos de disponibilidad y durabilidad incluso en caso de fallas.

En cuanto a la configuración de la replicación de objetos en Azure Storage, puedes utilizar el servicio de Azure Blob Storage object replication para copiar de forma asincrónica blobs en bloques de un contenedor de una cuenta de almacenamiento a otra. Para configurar la replicación de objetos, debes crear una política de replicación que especifique la cuenta de almacenamiento de origen y la cuenta de destino. Una política de replicación incluye una o más reglas que especifican un contenedor de origen y un contenedor de destino e indican qué blobs en bloques del contenedor de origen se replicarán.

Puedes configurar la replicación de objetos utilizando el portal de Azure, PowerShell, Azure CLI o una de las bibliotecas de cliente del proveedor de recursos de Azure Storage. Antes de configurar la replicación de objetos, asegúrate de que las cuentas de origen y destino existan y que la versión de blob esté habilitada para ambas cuentas. También es necesario que el cambio de blob esté habilitado para la cuenta de origen.

Las claves administradas por el cliente ofrecen más flexibilidad para administrar controles de acceso .

  • Módulo de seguridad de hardware (HSM) administrado por Azure Key Vault

Puede crear sus propias claves y almacenarlas en un almacén de claves o HSM administrado, o bien puede usar las API de Azure Key Vault para generarlas.

Configuración de las capas de almacenamiento

La documentación cubre la configuración de las capas de almacenamiento en Azure Blob Storage y Azure Files. En resumen, Azure Blob Storage admite tres tipos de capas de almacenamiento: Hot, Cool y Archive. Cada capa de almacenamiento tiene un costo diferente y está diseñada para satisfacer diferentes necesidades de almacenamiento. Por ejemplo, la capa Hot es adecuada para datos que se acceden con frecuencia, mientras que la capa Archive es adecuada para datos que se acceden con poca frecuencia y se deben conservar durante un período prolongado de tiempo.

Por otro lado, Azure Files admite dos tipos de capas de almacenamiento: Premium y Standard. La capa Premium es adecuada para cargas de trabajo que requieren un alto rendimiento y baja latencia, mientras que la capa Standard es adecuada para cargas de trabajo que no requieren un alto rendimiento y baja latencia.

  3. Selecciona Disabled para Soft delete en todos los recursos de archivos.
  4. Selecciona Save para confirmar la configuración de retención de datos.

En cuanto a las instantáneas, Azure Files proporciona la capacidad de tomar instantáneas de los recursos de archivos SMB. Las instantáneas de recursos de archivos capturan el estado del recurso en ese momento. Puedes utilizar las instantáneas de recursos de archivos para proteger contra errores de aplicación y corrupción de datos, así como para fines generales de copia de seguridad.

Cada capa de almacenamiento tiene un costo diferente y está diseñada para satisfacer diferentes necesidades de almacenamiento. Puedes utilizar la administración del ciclo de vida de los blobs para mover datos entre las capas de almacenamiento o para eliminar datos al final del ciclo de vida de los datos.

Para configurar la administración del ciclo de vida de los blobs, puedes seguir los siguientes pasos:

Por ejemplo, puedes crear una regla que mueva los datos a la capa Cool después de 30 días y a la capa Archive después de 90 días.
  4. Selecciona Save para confirmar la configuración de la regla de administración del ciclo de vida de los blobs.

  • Recursos: Tweets, usuarios, seguidores, etc.
  • Operaciones:Publicar un tweet (POST), obtener tweets de un usuario (GET), seguir a un usuario (POST), etc.
  • URI:https://api.twitter.com/
